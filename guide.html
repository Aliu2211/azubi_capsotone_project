Project Setup & Environment Configuration

Goal: Set up all tools, accounts, and repositories so the team can collaborate and deploy to AWS.

  1. Create IAM users with least-privilege permissions

    In the AWS Console, go to IAM → Users. Create one user per team member. Attach only the policies needed for their role (e.g. EC2, S3, CloudFront read/write). Never use the root account for daily work.

Web Application Development & Deployment

Goal: Build the documentation web app, deploy it on EC2 behind an ALB, and serve static assets from S3.

  1. Develop the web application

    Create index.html and guide.html (hosted on EC2 via git). Upload style.css, app.js, and screenshots to the S3 bucket. Reference assets with relative paths in HTML:

    <link rel="stylesheet" href="style.css">
<script src="app.js" defer></script>
<img src="Screenshots/phase1-iam-users.png" alt="...">

  2. Clone repo on EC2 and configure Apache

    cd /var/www/html
sudo git clone https://github.com/YOUR-ORG/YOUR-REPO.git .
sudo chown -R apache:apache /var/www/html

  3. Create Target Group

    EC2 → Target Groups → Create. Protocol HTTP, port 80. Register your EC2 instance. Set health check path to /.

    Step 3 — Target GroupScreenshots/phase2-alb-target-group.png
    Target Group with registered EC2 instances
    📷

    Screenshot needed: ALB Target Group

    Screenshots/phase2-alb-target-group.png
    Capture: EC2 → Target Groups → Targets tab. Show instance IDs and health status.

  4. Create Application Load Balancer

    Internet-facing ALB in public subnets. Listener: HTTP → redirect to HTTPS. HTTPS listener forwards to Target Group with ACM certificate.

    Step 4 — ALB ListenersScreenshots/phase2-alb-listener.png
    ALB listeners: HTTP redirect and HTTPS forward
    📷

    Screenshot needed: ALB Listeners

    Screenshots/phase2-alb-listener.png
    Capture: EC2 → Load Balancers → Listeners tab. Show HTTP:80 redirect rule and HTTPS:443 forward rule.

  5. Set up Auto Scaling Group

    Create a Launch Template from your EC2 instance. ASG min=2, max=4, desired=2. Attach the Target Group for automatic registration.

    Step 5 — Auto Scaling GroupScreenshots/phase2-asg.png
    ASG with launch template and desired capacity

    Screenshot needed: Auto Scaling Group

aws s3 cp app.js   s3://capstone-s3-343073438650-us-east-1-an/app.js   --profile capstone
aws s3 cp architecture.png s3://capstone-s3-343073438650-us-east-1-an/architecture.png --profile capstone
aws s3 sync Screenshots/ s3://capstone-s3-343073438650-us-east-1-an/Screenshots/ --profile capstone

    Confirm all registered targets show healthy in the Target Group before proceeding to CloudFront.

Architecture tipKeep HTML on EC2 (dynamic deploy via git) and static assets on S3 (CSS, JS, images). The browser loads them separately — CloudFront does not pull S3 content onto EC2.

Security Hardening & CloudFront Integration

Goal: Add CloudFront as the CDN layer, enforce HTTPS end-to-end, and lock down security groups and S3 access.

  1. Create CloudFront distribution

    Origin: your ALB DNS name. Viewer protocol policy: Redirect HTTP to HTTPS. Attach ACM certificate. Enable HTTP/2.

    Step 1 — CloudFront DistributionScreenshots/phase3-cloudfront-dist.png
    CloudFront distribution pointing to ALB origin
    📷

    Screenshot needed: CloudFront Distribution

    Screenshots/phase3-cloudfront-dist.png
    Capture: CloudFront → Distributions. Show distribution domain, status Enabled, and Origins tab with ALB DNS.

  2. Configure cache behaviors

    Default behavior caches dynamic HTML conservatively. Add a behavior for /static/* or S3 origin with longer TTL for CSS/JS/images.

    Step 2 — Cache BehaviorsScreenshots/phase3-cloudfront-behaviors.png
    Behaviors: HTTPS redirect and cache policies
    📷

    Screenshot needed: CloudFront Behaviors

    Screenshots/phase3-cloudfront-behaviors.png
    Capture: CloudFront → Behaviors tab. Show viewer protocol policy (Redirect HTTP to HTTPS) and cache policy settings.

  3. Restrict EC2 security groups

    Remove open HTTP/HTTPS from 0.0.0.0/0. Add inbound rules using the com.amazonaws.global.cloudfront.origin-facing managed prefix list only.

    Step 3 — Security GroupScreenshots/phase3-sg-cloudfront-only.png
    EC2 SG inbound: CloudFront managed prefix list only
    📷

    Screenshot needed: Security Group Lockdown

    Screenshots/phase3-sg-cloudfront-only.png
    Capture: EC2 → Security Groups → Inbound rules. Show HTTP/HTTPS from com.amazonaws.global.cloudfront.origin-facing prefix list only.

  4. Enable S3 Origin Access Control (OAC)

    Create OAC on the S3 bucket. Update bucket policy to allow only your CloudFront distribution. Block all public S3 access.

    Step 4 — S3 OACScreenshots/phase3-s3-oac.png
    S3 bucket policy allowing CloudFront OAC only
    📷

    Screenshot needed: S3 Origin Access Control

    Screenshots/phase3-s3-oac.png
    Capture: S3 → Bucket → Permissions. Show OAC configuration and bucket policy with CloudFront distribution ARN.

  5. Apply least-privilege IAM policies

    Review all IAM roles and policies. Remove unused permissions. Scope S3, EC2, and CloudFront actions to specific resource ARNs.

  6. Test the full HTTPS flow

    Browser → CloudFront URL (padlock) → ALB → EC2. Verify CSS and JS load from S3. Confirm EC2 is unreachable directly from the public internet.

    Step 6 — HTTPS Flow TestScreenshots/phase3-https-flow.png
    Browser address bar: padlock + CloudFront distribution URL
    📷

    Screenshot needed: HTTPS Flow Test

    Screenshots/phase3-https-flow.png
    Capture: Browser showing your CloudFront URL with HTTPS padlock icon and the site loaded correctly.
🔒
Security decisionLocking EC2 to CloudFront-only traffic means attackers cannot bypass the CDN and hit your instances directly. OAC ensures S3 assets are only served through CloudFront.

CI/CD Pipeline, Monitoring & Cost Management

Goal: Automate deployments with GitHub Actions, set up observability, and monitor Free Tier costs.

  1. Create GitHub Actions workflow

    On push to main, authenticate via OIDC, discover ASG instance IPs, SSH in, and run git pull + restart Apache.

    # .github/workflows/deploy.yml
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: aws-actions/configure-aws-credentials@v2
        with:
          role-to-assume: arn:aws:iam::ACCOUNT:role/Capstone-GitHub-Actions-Role
      # SSH deploy to all ASG instances...

    Step 2 — CloudWatch AlarmsScreenshots/phase4-cloudwatch-alarms.png

  3. Enable CloudWatch Logs

    Follow the instructions in the CloudWatch Logs guide to configure Apache access/error logs with 7-day retention. Install CloudWatch agent on EC2 if needed.
